Verdict du Council

Où le Council est d'accord

  • L'infrastructure est solide — Programme Anchor déployé, API complète, frontend connecté. Les rails sont posés.
  • L'agent routeur est la pièce manquante critique — Sans lui, le projet est un registry passif. Un aéroport sans avions.
  • Une démo live end-to-end est non-négociable pour un jury hackathon. 52 agents seedés manuellement ne prouvent rien.
  • Le scoring à 6 composantes est un vrai différenciateur — mais il doit être utilisé activement par le routeur.

Où le Council diverge

  • Cas d'usage vs. code — L'Outsider dit qu'il manque un cas d'usage concret. L'Executor dit que 4h de code SONT le cas d'usage. Résolution : choisir la tâche concrète AVANT de coder.
  • Ambition vs. réalité — L'Expansionniste veut présenter "Upwork autonome de l'IA". Le Contrarian dit que sans preuve, c'est du vent. Pour le hackathon : formuler l'ambition, démontrer un slice.

Angles morts détectés

  • Zéro tests Anchor = risque de crash en démo — Un bug en live tue le projet.
  • Le "pourquoi blockchain" n'est pas évident — Le routeur DOIT utiliser le score on-chain pour ses décisions, sinon un JSON suffirait.
  • Les agents ne font rien de visible — Il faut des tâches concrètes (traduction, résumé, audit) pas des hash abstraits.

La Recommandation

Implémenter un seul scénario hardcodé de bout en bout.
Un utilisateur soumet "Analyse ce texte et traduis-le en français". Le routeur lit le registry, sélectionne les agents par score, crée des escrows on-chain, les agents exécutent, l'escrow se libère. Tout visible en temps réel sur la page /demo.

Plan d'implémentation de l'Agent Routeur

  1. router-agent.ts (2h) — Keypair dédié, s'auto-enregistre dans le registry. Reçoit une mission, appelle GET /agents/search?capability=X, sélectionne par score, crée les escrows.
  2. Worker agents simulés (30min) — Scripts qui écoutent leurs escrows, "exécutent" (sleep + résultat), appellent POST /escrow/:id/submit puis release se fait automatiquement.
  3. Page /demo (1h) — Textarea mission → bouton → affichage temps réel de chaque étape (agents sélectionnés, escrows créés, résultats, paiements).
  4. Test Anchor minimal (30min) — Un seul test : create_escrow + submit_result + verify_and_release. Anti-crash.

Total estimé : ~4h de code

Perspectives des conseillers

Le Contrarian — "Une vitrine sans moteur"

L'agent routeur n'existe pas. 52 agents seeded ne prouvent rien. Zéro tests. 6 pages frontend en TODO. Sans une démo verticale complète, c'est un projet qui raconte une histoire, pas qui la prouve. Un seul flow parfait vaut mieux que 18 routes d'API sans moteur.

Le First Principles — "Les rails sans le train"

La seule chose qui compte : un agent peut-il déléguer du travail de façon trustless avec règlement automatique ? Renverser la priorité : la plus petite démonstration viable bat 10 000 lignes d'infrastructure. Un routeur minimaliste + une seule page de flux temps réel.

L'Expansionniste — "Le Upwork autonome de l'IA"

Ce n'est pas un registry — c'est un marché de travail on-chain. Le scoring est un signal de qualité vérifiable. Le routeur devrait enchérir par score. Mieux : le routeur s'auto-enregistre comme agent. Recursivité totale. L'infrastructure est à 70%.

L'Outsider — "Comptabilité pour robots imaginaires"

Où sont les tâches concrètes ? Pourquoi blockchain ? Qui décide qu'une tâche est complétée ? Il manque une démo 90 secondes avec une tâche visible. L'absence d'un cas d'usage concret est plus grave que l'absence du routeur.

L'Executor — "4h de coding, pas plus"

Step 1: router-agent.ts (2h) — keypair, sign, search agents, create escrows, poll. Step 2: scénario hardcodé (30min) — 3 subtasks, sleep+submit. Step 3: page /demo streaming (1h). Step 4: check balances. Start with the router, the rest follows.

La première chose à faire

Maintenant

Créer router-agent.ts avec uniquement ceci : lire le registry Anchor via l'API, trouver deux agents avec des capabilities différentes, afficher leurs scores dans la console. Pas d'escrow, pas de frontend. Juste la preuve que le routeur peut lire intelligemment le registry et prendre une décision basée sur le score on-chain.
